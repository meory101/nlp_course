Section 2 The Argument in Five Lines

Long agent runs fail because a tiny early mistake snowballs through the rest of the run. (Papers A, B)

People can detect drift — but only after it is already happening, or only on single standalone questions, never per-step inside a run. (Papers D, E)

Repair works when given an accurate pointer — but every test did it post-mortem, after the run already failed; a wrong pointer is worse than silence. (Papers I, K)

Nobody fires an early alarm during the run to prevent the failure — or measures whether firing earlier saves more runs.

And nobody has measured whether that early alarm actually helps the agent recover — neither with a label/verifier nor without one (intrinsic).

That is my project: detect drift early → measure lead time → fire a live alarm and test recovery both with and without a verifier, before the run is lost.

Section 3 The Six Papers We Read

— RQ1 · MEASURE —
A
The Illusion of Diminishing Returns: Measuring Long-Horizon Execution in LLMs
ICLR 2026 · arXiv 2509.09677
Per-step error rises as a run grows — the model imitates its own past mistakes (self-conditioning). Failure is execution, not reasoning.

MODELS (per experiment)

  • 1 · Illusion / horizon: full suite — GPT-5, Claude-4-Sonnet, Grok 4, Gemini-2.5-Pro, Kimi K2, Qwen3-235B, DeepSeek-R1
  • 2 · Execution not reasoning: Qwen3 + Gemma3 only
  • 3 · Self-conditioning: Qwen3, Gemma3 (+ Kimi-K2, DeepSeek-V3)

BENCHMARK / DATA

  • Dictionary-Sum — synthetic; free labels every step
  • Data ✓ + task generator ✓ + fake-history injection code ✓
  • github.com/long-horizon-execution/measuring-execution
  • HF: arvindh75/Long-Horizon-Execution

THE GAP

  • Only measures drift — passive, after the run
  • Never raises a live alarm or intervenes → my RQ2/RQ3
  • (sterile lab = the trade-off that gives free labels, not a flaw)

WHAT I USE

  • The lab + free per-step labels
  • Scrub repair: their injection run backwards — erase the poison vs correct over it (RQ3/RQ4)

K
Find-vs-Fix — LLMs Cannot Find Reasoning Errors, but Can Correct Them Given the Error Location
ACL 2024 Findings · arXiv 2311.08516 · Google / Cambridge
Models find their error step only ~53% of the time — but given the true location, backtracking repair gains +18…+44 points.

MODELS

  • Traces: PaLM 2
  • Finders: GPT-4 (52.9%),
  • GPT-4-Turbo, GPT-3.5, Gemini Pro

BENCHMARK / DATA

  • BIG-Bench Mistake — 2,186 traces, GOLD first-error-step labels
  • Apache 2.0 ✓ + prompts (detector not released)
  • github.com/WHGTyen/BIG-Bench-Mistake

THE GAP

  • Pointer = post-mortem oracle, given after the run ended
  • Recovery vs earliness never measured

WHAT I USE

  • Backtracking = my repair (alarm replaces oracle)
  • Target: ~60–70% detector accuracy suffices
  • Their data = offline test bed: 2,186 traces with human gold error-step labels → replay prefixes, check my signal rises before the gold step = lead time vs ground truth
  • caveat: frozen traces → only passive signals; active probes (scrub / ask-twice) need live runs in A's lab
